## 1. 프로젝트 개요

이 서비스는 1인 가구를 중심으로 한 사용자가 자신이 보유한 식재료를 등록하고, 이를 기반으로 실제로 만들 수 있는 요리 또는 재료를 조금만 더 구매하면 만들 수 있는 요리를 추천받는 웹/모바일 서비스이다. 레시피 데이터는 식품의약품안전처의 공공 레시피 데이터를 **CSV 파일로 다운로드하여 1회성으로 DB에 적재**해 사용하며(실시간 API 호출 방식이 아님), 사용자의 냉장고 데이터는 사용자가 직접 검색·등록한다. 기존 레시피 앱들이 재료 목록만 나열하고 "내가 진짜 만들 수 있는지"는 알려주지 않는다는 문제의식에서 출발했으며, 보유 재료와 레시피 필요 재료를 자동으로 대조하여 실행 가능성 중심의 추천을 제공하는 것이 핵심 차별점이다. 여기에 더해 레시피 완성 처리에 따른 재료 자동 차감, 즐겨찾기·최근 조회·소진 통계 같은 재방문 유도 기능, 유통기한 관리, 레시피별 영양 정보 제공까지 포함해 "오늘 뭐 해먹지"를 넘어 재료 소진과 가벼운 식단 관리까지 지원하는 것을 목표로 한다.

## 2. 타깃 사용자 분석

### 페르소나 1: "오늘 뭐 해먹지" 고민하는 1인 가구 직장인

- **기본 정보**: 20~30대, 자취 3년 차, 평일 퇴근 후 요리 여부를 그날그날 정하는 편
- **핵심 니즈**:
  - 냉장고에 있는 재료로 뭘 만들 수 있는지 매번 검색하기 귀찮음
  - 장을 보러 가기 전에 "딱 하나만 더 사면 되는" 요리를 알고 싶음
  - 재료를 낭비하지 않고 소진하고 싶음
  - 자주 해먹는 요리, 자주 떨어지는 재료를 스스로 파악하고 싶음
- **주요 페인포인트**: 일반 레시피 앱은 재료 목록만 보여줄 뿐, 내가 지금 뭘 갖고 있는지와 대조해주지 않고, 요리 후 재료 현황을 다시 손으로 갱신해야 함

> 가정: 사용자 요청에 페르소나가 명시적으로 2개 제시되지 않아, 제공된 시나리오를 기반으로 페르소나 1개를 상세화하여 작성함.

## 4. 사용자 플로우

### 4.1 주요 시나리오: 신규 사용자의 첫 이용

1. **회원가입/로그인**: 사용자가 이메일로 회원가입 후 로그인한다.
2. **식재료 등록**: 검색창에 재료명을 입력하면 부분 문자열 매칭으로 후보가 노출된다(예: "파" 검색 시 양파/대파/실파/쪽파/파슬리 등). 원하는 이름이 없으면 목록 최하단의 "직접 추가하기"로 검색어 그대로 등록할 수 있다.
3. **나의 냉장고 화면 진입**: 등록이 완료되면 '나의 냉장고' 화면으로 이동하며, 등록된 재료가 카테고리별로 그룹핑되어 표시된다. 유통기한을 입력하면 카테고리 기준 기본값이 자동 제안된다.
4. **재료 추가**: 화면의 '+' 버튼으로 언제든 재료를 추가 등록할 수 있다.
5. **레시피 추천 탭 이동**: 웹은 상단 네비게이션, 모바일은 하단 탭에서 '레시피 추천'으로 이동한다.
6. **대시보드 확인**: 보유 재료로 완성 가능한 요리가 우선 노출되며, 필터로 "완전 제작 가능"/"일부 보충 필요"를 선택해 볼 수 있다.
7. **레시피 상세 확인**: 카드를 클릭하면 상세 화면으로 이동해 보유/부족 재료, 조리 순서, 영양 정보를 확인하고, 즐겨찾기에 추가할 수 있다.
8. **레시피 완성 처리**: 요리를 마친 뒤 "이 요리 완성했어요" 버튼을 누르면 사용한 재료가 나의 냉장고에서 자동으로 미보유 처리되고, 완성 기록이 저장된다.
9. **마이페이지 확인**: 최근 조회한 요리, 즐겨찾기한 요리, 자주 만든 요리/자주 소진된 재료 통계를 확인할 수 있다.

### 4.2 재방문 시나리오

1. 로그인 → '나의 냉장고' 화면에서 현재 보유 현황 확인 (유통기한 임박 재료 우선 확인)
2. 사용한 재료가 있다면 완성 버튼을 통해 자동 반영되거나, 직접 체크 해제
3. '레시피 추천' 탭에서 갱신된 재료 기준으로 다시 추천 확인
4. '마이페이지'에서 그간의 소진/조리 패턴을 통계로 확인

## 5. 기술 스택

| 영역 | 선택 기술 | 선택 이유 |
|---|---|---|
| 프론트엔드 + 백엔드 | Next.js (App Router, 풀스택) | 웹/모바일 반응형 대응이 용이하고, API Route로 별도 백엔드 서버 없이 인증/데이터 로직 처리 가능 |
| 인증 | Supabase Auth | 이메일 기반 회원가입/로그인을 빠르게 구현 가능 |
| 데이터베이스 | Supabase (PostgreSQL) | 관계형 데이터(사용자-재료-레시피-통계)에 적합 |
| 레시피 데이터 소스 | 식품의약품안전처 공공 레시피 데이터(CSV) | **API 실시간 호출이 아닌 CSV 1회성 적재 방식으로 확정.** 적재 스크립트를 수동 재실행하지 않는 한 자동 갱신되지 않음 |
| 배포 | Vercel | Next.js와의 호환성, 간편한 CI/CD |

> 참고: 프론트/백엔드를 물리적으로 분리하는 아키텍처도 논의되었으나, 최종적으로 본 PRD 기준(Next.js 풀스택 단일 프로젝트)으로 진행하는 것으로 확정함. BACKEND.md/FRONTEND.md는 같은 코드베이스 내에서 책임 영역을 문서상 구분한 것이며, 별도 서버로 분리 배포하지 않음.

**ingredient_consumption_log**: id, user_id, ingredient_name, consumed_at (F14 통계 기반)

### 6.2 관계 설명

- 1명의 `user`는 여러 `user_fridge`, `user_favorites`, `recipe_views`, `recipe_completions` 레코드를 가진다 (1:N)
- 레시피 완성 처리(F11) 시: `recipe_completions`에 기록 남기고, 해당 레시피의 `parsed_ingredients` 중 사용자가 보유했던 재료를 `user_fridge.is_owned = false`로 갱신하며, 각 재료를 `ingredient_consumption_log`에 기록
- 마이페이지 통계(F14)는 `recipe_completions`/`ingredient_consumption_log`를 그룹핑·카운트하여 산출 (별도 집계 테이블 없이 조회 시점에 계산하는 것으로 가정)

> 상세 스키마, API 엔드포인트, 비즈니스 로직은 BACKEND.md를 참고. 화면별 상태 관리, 컴포넌트 구성은 FRONTEND.md를 참고.

## 7. 개발 우선순위

### MVP 범위 (P0)
F1~F9 (회원가입/로그인, 검색 기반 재료 등록, 나의 냉장고, 기본 조미료 자동 처리, 레시피 추천 대시보드 및 필터, 재료 매칭, 레시피 상세, 플랫폼별 네비게이션)

F10~F16 (유통기한 관리, 레시피 완성 처리, 즐겨찾기, 최근 조회, 마이페이지 통계, 레시피별 영양 정보, 하루 섭취량 트래킹)

F17~F20 (목표 기반 맞춤 식단 필터, 사진/OCR 재료 인식, 바코드 스캔 등록, 네이티브 앱 전환)
