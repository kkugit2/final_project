# PRD: 냉장고 재료 기반 레시피 추천 서비스

- 문서 버전: v1.0
- 작성일: 2026-07-16
- 작성자: 기획 담당 (AI)

> 가정: 본 문서에서 "프로젝트 루트 디렉토리"는 실제 코드 저장소 구조가 부재한 환경 특성상 산출물 폴더(`/mnt/user-data/outputs`)로 대체하여 저장함.

---

## 1. 프로젝트 개요


이 서비스는 1인 가구를 중심으로 한 사용자가 자신이 보유한 식재료를 등록하고, 이를 기반으로 실제로 만들 수 있는 요리 또는 재료를 조금만 더 구매하면 만들 수 있는 요리를 추천받는 웹/모바일 서비스이다. 레시피 데이터는 식품의약품안전처의 공공 레시피 데이터(조리식품 레시피 DB, 1,146개 레시피)를 CSV 파일로 사전 적재하여 활용하며, 사용자의 냉장고 데이터는 사용자가 직접 등록한다. 기존 레시피 앱들이 재료 목록만 나열하고 "내가 진짜 만들 수 있는지"는 알려주지 않는다는 문제의식에서 출발했으며, 보유 재료와 레시피 필요 재료를 자동으로 대조하여 실행 가능성 중심의 추천을 제공하는 것이 핵심 차별점이다. MVP는 텍스트 기반 재료 등록과 회원별 냉장고 관리, 레시피 매칭 대시보드에 집중하고, 유통기한 관리·사진 인식 등은 이후 단계로 분리한다.

## 2. 타깃 사용자 분석

### 페르소나 1: "오늘 뭐 해먹지" 고민하는 1인 가구 직장인

- **기본 정보**: 20~30대, 자취 3년 차, 평일 퇴근 후 요리 여부를 그날그날 정하는 편
- **핵심 니즈**:
  - 냉장고에 있는 재료로 뭘 만들 수 있는지 매번 검색하기 귀찮음
  - 장을 보러 가기 전에 "딱 하나만 더 사면 되는" 요리를 알고 싶음
  - 재료를 낭비하지 않고 소진하고 싶음
- **주요 페인포인트**: 일반 레시피 앱은 재료 목록만 보여줄 뿐, 내가 지금 뭘 갖고 있는지와 대조해주지 않음

> 가정: 사용자 요청에 페르소나가 명시적으로 2개 제시되지 않아, 제공된 시나리오(1인 가구가 보유 재료로 요리를 고민)를 기반으로 페르소나 1개를 상세화하여 작성함. 추가 페르소나(예: 자취 대학생, 맞벌이 가정 등)가 필요하면 확장 가능.

## 4. 사용자 플로우

### 4.1 주요 시나리오: 신규 사용자의 첫 이용

1. **회원가입/로그인**: 사용자가 이메일로 회원가입 후 로그인한다.
2. **식재료 등록**: 로그인 직후(또는 온보딩 화면에서) 식재료 검색창에 재료명을 입력하면, 사전 등록된 식재료 DB에서 자동완성 형태로 검색 결과가 제시된다. 사용자는 보유한 재료를 체크(보유 여부 토글)하여 등록한다.
3. **나의 냉장고 화면 진입**: 등록이 완료되면 '나의 냉장고' 화면으로 이동하며, 등록된 재료가 카테고리별(육류/채소/유제품/조미료 등)로 그룹핑되어 한눈에 표시된다.
4. **재료 추가**: 화면 우측 하단(또는 상단) '+' 버튼을 눌러 언제든 재료를 추가 등록할 수 있다.
5. **레시피 추천 탭 이동**: 하단 탭(또는 상단 내비게이션)에서 '레시피 추천' 탭으로 이동한다.
6. **대시보드 확인**: 현재 보유 재료로 완성 가능한 요리가 상단에, 일부 재료만 보충하면 만들 수 있는 요리가 그다음에 카드 형태로 정렬되어 노출된다. 사용자는 필터를 통해 "완전 제작 가능" 또는 "일부 보충 필요" 중 하나만 선택해서 볼 수도 있다.
7. **레시피 상세 확인**: 특정 요리 카드를 클릭하면 상세 레시피 화면으로 이동하여, 전체 재료 목록(보유/부족 표시 포함)과 조리 순서를 확인한다.

> 가정: 화면 전환은 하단 탭 내비게이션(나의 냉장고 / 레시피 추천 / 마이페이지) 구조로 가정하고 작성함. 실제 IA는 UI/UX 디자이너 단계에서 구체화 필요.

### 4.2 재방문 시나리오

1. 로그인 → '나의 냉장고' 화면에서 현재 보유 현황 확인
2. 사용한 재료가 있다면 체크 해제(보유 여부 OFF), 새로 산 재료는 검색 후 추가
3. '레시피 추천' 탭에서 갱신된 재료 기준으로 다시 추천 확인

## 5. 기술 스택

| 영역 | 선택 기술 | 선택 이유 |
|---|---|---|
| 프론트엔드 + 백엔드 | Next.js (App Router, 풀스택) | 웹/모바일 반응형 대응이 용이하고, API Route로 별도 백엔드 서버 없이 인증 로직 처리 가능. SSR/SSG로 초기 로딩 성능 확보 |
| 인증 | Supabase Auth | 이메일 기반 회원가입/로그인을 빠르게 구현 가능, Next.js와 연동 라이브러리 지원 우수 |
| 데이터베이스 | Supabase (PostgreSQL) | 관계형 데이터(사용자-재료-레시피 매칭)에 적합, Auth와 통합 관리로 개발 속도 향상 |
| 레시피 데이터 | CSV 파일 (food_recipes_data.csv) | 식약처 공공 데이터 1,146개 레시피 사전 적재, 실시간 API 호출 불필요 |
| 배포 | Vercel (Next.js 배포에 최적화) | Next.js와의 호환성, 간편한 CI/CD |

> 가정: 배포 플랫폼은 사용자 요청에 명시되지 않아 Next.js와 궁합이 좋은 Vercel로 가정함. 필요시 변경 가능.

## 6. 데이터 구조

### 6.1 핵심 엔티티

**users** (Supabase Auth 기본 제공 테이블 확장)
| 필드 | 타입 | 설명 |
|---|---|---|
| id | uuid (PK) | Supabase Auth 사용자 ID |
| email | text | 로그인 이메일 |
| created_at | timestamp | 가입일 |

**ingredients_master** (사전 등록된 식재료 검색 DB)
| 필드 | 타입 | 설명 |
|---|---|---|
| id | uuid (PK) | 재료 고유 ID |
| name | text | 재료명 (예: 돼지고기, 양파) |
| category | text | 카테고리 (육류/채소/유제품/조미료 등) |
| synonyms | text[] | 동의어 목록 (예: 대파 → 파, 쪽파) — 레시피 매칭용 |
| is_basic_seasoning | boolean | 기본 조미료 자동 처리 여부 (true면 매칭 시 항상 보유로 간주) |

**user_fridge** (사용자별 보유 재료)
| 필드 | 타입 | 설명 |
|---|---|---|
| id | uuid (PK) | 레코드 ID |
| user_id | uuid (FK → users.id) | 소유 사용자 |
| ingredient_id | uuid (FK → ingredients_master.id) | 등록된 재료 |
| is_owned | boolean | 보유 여부 |
| updated_at | timestamp | 최종 수정일 |

**recipes_cache** (CSV 파일 기반 레시피 데이터)
| 필드 | 타입 | 설명 |
|---|---|---|
| id | text (PK) | CSV 데이터의 RCP_SEQ (레시피 고유 ID) |
| name | text | 요리명 (RCP_NM) |
| raw_ingredients_text | text | 원본 재료 텍스트 (RCP_PARTS_DTLS) |
| parsed_ingredients | jsonb | 파싱 후 정규화된 재료명 배열 |
| cooking_steps | jsonb | 조리 순서 (MANUAL01~MANUAL20) |
| image_url | text | 완성 이미지 URL |
| loaded_at | timestamp | CSV 데이터 로드 시각 |

> CSV 파일(food_recipes_data.csv)에서 사전에 적재된 1,146개 레시피 데이터를 recipes_cache 테이블에 로드합니다. 실시간 API 호출 불필요.

### 6.2 관계 설명

- 1명의 `user`는 여러 개의 `user_fridge` 레코드를 가진다 (1:N)
- `user_fridge`는 `ingredients_master`를 참조하여 실제 재료 정보를 가져온다 (N:1)
- 레시피 매칭 시: `recipes_cache.parsed_ingredients`와 사용자의 `user_fridge(is_owned=true)` + `ingredients_master(is_basic_seasoning=true)` 재료를 대조하여 일치/불일치 항목을 계산한다 (매칭은 저장되지 않고 조회 시점에 실시간 계산되는 것으로 가정)
