## 2. 화면별 스펙

### 2.1 나의 냉장고 (`/fridge`)
**목록 화면**
- 카테고리별 섹션(채소/육류/유제품/조미료/새로 추가됨 등), 섹션 헤더 + 아이템 리스트
- 각 아이템: 토글(체크박스 아이콘, 시각적 크기 24×24px이나 **탭 가능 영역은 패딩 포함 최소 44×44px** 확보 — 4장 터치 대상 기준 준수) + 재료명, 클릭 시 즉시 보유 여부 반전
- 하단(또는 리스트 끝) "재료 추가" 버튼(Primary, Steel Blue)

**재료 추가 화면 (검색 기반, 브라우징 아님)**
- 진입 시 검색창만 노출, 결과 리스트는 비어있는 상태("검색어를 입력하면 재료를 찾을 수 있어요")
- 타이핑 즉시 부분 문자열 매칭 결과 노출 (디바운스 150~200ms 권장)
- 검색 결과 유무와 무관하게 리스트 최하단에 **`"{검색어}" 직접 추가하기`** 항목 항상 노출
- 결과 없을 때: "검색 결과가 없어요. 아래에서 직접 추가할 수 있어요" 안내 문구 + 직접 추가 항목
- 항목 클릭으로 다중 선택(선택 시 배경 `#DCE6EC` + 체크 아이콘), 하단 "선택한 재료 추가 (N)" 버튼으로 일괄 confirm
- confirm 시 fridge 재료 추가 Server Action 호출 (마스터 재료는 `ingredient_id`, 커스텀은 `custom_name`으로 구분 전송, 5장 참고)

**유통기한 (P1)**
- 재료 등록 시 카테고리 기준 기본 유통기한을 자동 계산해 표시, 사용자가 직접 수정 가능
- 목록에서 유통기한 임박 재료는 시각적으로 강조(별도 색상 추가 없이, 텍스트 굵기/작은 라벨로 구분 — 가이드라인의 "색상 최소화" 원칙 유지)

### 2.2 레시피 추천 (`/recipes`)
**목록 화면**
- 상단 필터: 전체 / 완전 보유 / 일부 부족 (Steel Blue 채움 vs 윤곽선으로 활성 상태 구분)
- 카드 그리드: 모바일 1~2열, 태블릿 2열, 데스크톱 3열
- 카드 구성: 사진(16:9) + 요리명 + 매칭 배지(완전 매칭: 채움 배지 / 일부 부족: 윤곽선 배지)
  - 이미지는 `next/image`로 렌더링 (`next.config.js`의 `images.remotePatterns`에 `www.foodsafetykorea.go.kr` 등록 필요 — 원본이 HTTP여도 next/image가 서버에서 최적화·재서빙하므로 브라우저 혼합 콘텐츠 경고 없음, BACKEND-INTEGRATED.md 3.2 참고)
  - `image_url`이 null이면 플레이스홀더(회색 배경) 표시
- 정렬 기본값: 완전 매칭 우선

**상세 화면**
- 뒤로가기 버튼, 요리명, 즐겨찾기 아이콘(하트, 클릭 시 즐겨찾기 추가/해제 Server Action 호출, 5장 참고)
- **즐겨찾기 아이콘 근처에 키워드 태그 2개 노출**: ① 요리 종류(`recipes.category` — 반찬/일품/후식/밥/국&찌개/기타), ② 조리법(`recipes.cooking_method` — 굽기/끓이기/볶기/찌기/튀기기/기타). 뉴트럴 톤 태그(`IngredientTag`와 동일한 스타일, 배경 `#E8ECEE`)로 표시해 매칭 배지(Steel Blue 채움/윤곽선)와 시각적으로 구분
- 매칭 배지, 보유 재료 태그 목록, 부족 재료 태그 목록(부족 재료 없으면 섹션 자체 숨김)
- 조리 순서(번호 목록)
- 영양 정보 표시 (P1, F15): 1인분 기준 칼로리/탄단지/나트륨 — `recipes` 테이블의 nutrition 필드 그대로 노출
- 하단 "이 요리 완성했어요" 버튼 → 완성 처리 Server Action 호출(5장 참고), 성공 시 버튼 텍스트 일시적으로 "완성 기록 저장됨"으로 피드백 후 원복
- 진입 시 조회 기록 Server Action 자동 호출 (최근 조회 기록, 5장 참고)

### 2.3 마이페이지 (`/mypage`)
- **최근 조회한 요리**: 가로 스크롤 카드(최대 5개), 클릭 시 레시피 상세로 이동
- **즐겨찾기한 요리**: 세로 리스트, 하트 아이콘으로 즐겨찾기 해제 가능
- **이번 달 인사이트**: "가장 자주 만든 요리 TOP 3", "가장 자주 소진된 재료 TOP 3" — 텍스트+횟수 위주(그래프 사용 안 함)
- 데이터 없을 때 각 섹션에 빈 상태 문구 표시 ("아직 조회한 요리가 없어요" 등)
- **하루 섭취량 트래킹 (P1, F16)**: 오늘 완성한 요리들의 칼로리 합산 표시
- **목표 기반 필터 (P2, F17)**: `user_profiles.goal` 설정 시 레시피 추천 화면에 필터 옵션 추가. 의학적 식이 제한은 다루지 않는다는 안내 문구 필수 노출

## 3. 상태 관리

- **초기 데이터 로드**: App Router의 Server Component에서 직접 Supabase를 조회해 서버에서 렌더링 (냉장고 목록, 레시피 목록, 마이페이지 요약의 최초 화면은 클라이언트 fetch 없이 서버에서 채워서 내려줌 — 로딩 스피너와 워터폴 요청을 줄이는 최신 App Router 권장 패턴)
- **사용자 액션(변경 작업)**: 재료 추가/토글, 즐겨찾기, 완성 처리 등은 **Server Action**으로 구현하고, 클라이언트에서는 `useOptimistic`으로 즉각 반영 후 서버 응답으로 확정 — 매번 API Route를 왕복하는 것보다 코드가 단순해지고 타입도 자연스럽게 공유됨
- **클라이언트에서 추가 재검증이 필요한 경우**(예: 검색 결과처럼 입력마다 바뀌는 데이터): SWR 또는 React Query 사용 가능. 다만 App Router 환경에서는 Server Action + Server Component 조합만으로 충분한 화면이 많아, 꼭 필요한 곳에만 제한적으로 도입
- **로컬 UI 상태**(검색어 입력값, 선택된 필터, 모달 열림 여부 등): 컴포넌트 로컬 `useState`
- **전역으로 공유해야 하는 상태**(로그인 사용자 정보): Supabase Auth 세션을 Context로 감싸 최상위에서 제공

> 가정: 별도 전역 상태 라이브러리(Redux, Zustand 등)는 이 서비스 규모에서 불필요하다고 판단하여 도입하지 않는 것으로 가정함. 추후 상태가 복잡해지면 재검토.
> 변경 메모: 기존에는 모든 서버 상태를 React Query/SWR로 가져오는 것을 기본값으로 제안했으나, App Router 도입 후 실무에서 자리잡은 "Server Component로 최초 로드 + Server Action으로 변경"이 코드량이 적고 워터폴이 없어 이를 기본 전략으로 변경함.

## 4. 디자인 시스템 요약 (상세는 UI-UX-Guideline.md 참고)

- **색상**: Steel Blue(`#4A7C9E`) 단일 액센트 + 아이스 그레이 뉴트럴 톤. 상태 구분은 색상이 아니라 채움/윤곽선/톤 농도로 표현
- **타이포그래피**: Inter, 8px 기준 그리드
- **터치 대상**: 최소 44×44px
- **명도 대비**: 본문/버튼 4.5:1 이상 목표 (일부 톤은 경계값이므로 실제 구현 시 WebAIM 등으로 재검증 권장)

## 5. 기능 연동 매핑 (BACKEND-INTEGRATED.md 5장과 동일 기준)

> 아래 표는 각 화면이 필요로 하는 논리적 작업 단위입니다. 초기 데이터 조회는 Server Component에서 직접 처리하고, 변경 작업은 Server Action으로 구현하는 것을 기본으로 하되(3장 참고), 팀 상황에 따라 전통적인 REST API Route로 구현해도 동일하게 동작합니다.

| 화면/기능 | 기능 단위 | 기본 구현 형태 |
|---|---|---|
| 나의 냉장고 목록 | fridge 목록 조회 | Server Component |
| 재료 검색 | GET /api/ingredients/search | API Route |
| 재료 추가 | fridge 재료 추가 | Server Action |
| 재료 보유 토글/유통기한 수정 | fridge 항목 수정 | Server Action |
| 레시피 목록 (필터 포함) | recipes 목록 조회 | Server Component |
| 레시피 상세 | recipes 상세 조회 | Server Component |
| 레시피 조회 기록 | recipes 조회 기록 | Server Action |
| 즐겨찾기 추가/해제 | recipes 즐겨찾기 추가/해제 | Server Action |
| 레시피 완성 처리 | recipes 완성 처리 | Server Action |
| 마이페이지 요약 | mypage 요약 조회 | Server Component |
